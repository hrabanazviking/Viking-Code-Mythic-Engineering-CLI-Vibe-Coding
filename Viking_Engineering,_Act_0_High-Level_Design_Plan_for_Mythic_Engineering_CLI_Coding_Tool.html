Introduction: From Chaos to Order

In the realm of AI engineering, a repository called "Viking" is mired in a classic case of "growing pains."

Built by a group of energetic "Viking" programmers, it is filled with experimental code, diverse configuration files, and a series of powerful tools capable of executing independent tasks. However, as ambitions have swelled, the system has gradually evolved into a chaotic collection of spaghetti structures, isolated code fragments, and tangled logic.

The Essence of the Problem

The repository is littered with uncleaned temporary files and arbitrary modifications to global state, like a battlefield after a fierce raid. The root of the problem is not a lack of code, but an absence of architecture.

The current predicament stems from inevitable friction when the "Viking" spirit is unleashed without discipline or foresight. Viking expeditions are legendary, but their ability to return home and build the foundation of an era relied not only on the passion for plunder, but on masterful shipbuilding, a deep understanding of currents, and a clear division of roles among the crew.

Mythic Narrative

Create a unified, meaningful design story for the system that transcends mere code functionality

Domain-Driven Design

Partition complex problem spaces into bounded, contextual domains

Ambient Developer Experience

Make the CLI feel like epic equipment in the eyes of developers

This report is not a typical to-do list. It is a strategic blueprint aimed at reshaping the "Viking" project into a robust, coherent, and delightful command-line interface in the eyes of developers.

2.3 Domain Laws and Invariant Constraints

Tool Summoning Domain Laws

  • Invariant 1: Tools must pass strict validation before being added to the Registry
  • Business Rule 1: All declared dependencies must be satisfied before execution
  • Error Handling Pact: Must return a structured ExecutionResult object on failure

Configuration Domain Laws

  • Invariant 1: Secrets must be stored encrypted in memory, plaintext values never logged
  • Business Rule 1: Configuration merging must follow a well-defined priority order
  • Validation Rule: All config items must be validated against a predefined JSON Schema

Project Domain Laws

  • Invariant 1: Project root directory must physically exist in the filesystem
  • Business Rule 1: New workspace creation must generate a Manifest file with necessary metadata
  • Lifecycle Rule: All state changes must be recorded as traceable ProjectEvents

Query & Search Domain Laws

  • Invariant 1: Empty query cannot return a wildcard query matching all results
  • Business Rule 1: Search results must include relevance scores and highlight metadata
  • Pagination Rule: Must support pagination with complete pagination info

III. High-Level Python Architecture Blueprint

3.1 System Architecture Overview

Hexagonal Architecture (Ports & Adapters)

Domain Layer

Core domain models & business logic

↑↓
Application Layer

Use case coordination & interface definitions

↑↓
Infrastructure Layer

External implementations & adapters

Dependency Rule

Dependencies point strictly inward: outer layers depend on inner layers, and inner layers know nothing of the outer layers. This makes the system extremely flexible, testable, and maintainable.

3.2 Mythic Core (mythic_core/)

Domain Models Package (domain/)

Shared Kernel (_shared/)
  • • Entity: Base class with unique ID and version
  • • ValueObject: Immutable value object base class
  • • AggregateRoot: Aggregate root base class
  • • DomainEvent: Domain event base class
  • • Result[T]: Generic result object
Tool Summoning Domain (tool_summoning/)
  • • Tool: Tool entity
  • • ToolMetadata: Tool metadata value object
  • • ToolService: Tool execution service
  • • ToolValidator: Tool validator

Application Services Package (application/)

Tool Application Service

Provides high-level use case interface SummonToolUseCase

  • • execute(command) → Result[ToolResult]
  • • Coordinates domain layer services
  • • Publishes domain events
Other Application Services
  • • SearchUseCase
  • • ConfigureUseCase
  • • ProjectUseCase

Error Handling Design

VikingError is the base class for all custom errors, providing structured error_code, message, and context.

The Result[T] generic object is used for service method returns, forcing callers to perform explicit error checking and avoiding exception-driven spaghetti code.

3.4 Tool Covenant (tool_covenant/)

viking.yml Manifest Schema

The tool manifest file is like the "blueprint" a blacksmith uses to forge a weapon, specifying the tool's form, capabilities, and usage in detail.

Core Configuration Items
  • tool: Tool identity metadata (id, name, version)
  • requirements: Dependencies & environment config
  • input: Input contract (JSON Schema)
  • execution: Execution logic definition
Extended Configuration Items
  • output: Output contract (JSON Schema)
  • ui: UI prompts & help information
  • docker: Containerized execution config
  • inline: Inline script support

IV. Refactoring Strategy for Transformation

4.1 Transitioning from Existing Code

Strangler Fig Pattern

Gradually and incrementally replace old code with the new architecture, ensuring the system remains functional throughout the refactoring process.

Step 1: Establish a Safety Net

Create integration tests covering core functionality as a "safety net" during refactoring.

Step 2: Extract Core Domain

Identify core business logic from existing code, extract into mythic_core/domain/ modules.

Step 3: Create an Anti-Corruption Layer

Provide translation and isolation between new and old code to prevent "corruption" spread.

Step 4: Parallel Development

Adopt feature freeze or parallel development strategies to ensure refactoring and new feature development don't interfere.

4.2 Building Pillars and Abstraction Layers

AI-Assisted Architecture Implementation Workflow

1. Define Core Interfaces

Turn application service interfaces into executable Python abstract base classes

2. AI-Assisted Generation

Leverage AI to create initial implementation adapters for interfaces

3. Constraint Testing

Write behavior-driven tests defining the contracts adapters must satisfy

4. AI Iterative Refinement

Guide AI to refine code under the "tests as law" constraint

New Human-AI Collaboration Model

Humans become the system's architects and quality assurance officers, while AI becomes a powerful but rule-bound "executive agent."

Human Role
  • • Set goals and constraints
  • • Define interfaces and tests
  • • Review results and decisions
AI Role
  • • Code generation and modification
  • • Handle implementation details
  • • Rapid iteration and optimization

4.3 Ensuring Long-Term Consistency

Architecture Decision Records

Create ADR documents for every major architectural decision, recording context and rationale.

Provides "institutional memory" for future maintainers and AI agents

Continuous Integration Guardian

Use tools like import-linter to automate architecture constraint checks.

Commits violating architectural rules are automatically rejected from merging

Developer Onboarding Optimization

A clear architecture blueprint provides excellent teachability.

New developers understand system structure within days
