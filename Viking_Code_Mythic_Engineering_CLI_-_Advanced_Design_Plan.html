Introduction

This report defines "Viking Code," a command-line interface (CLI) tool designed to be the ultimate command center for "Mythic Engineering." It is not just another simple code generator, but a powerful, terminal-centric platform designed to harness and direct the collective action of multiple AI agents, meeting the increasingly complex demands of modern AI-driven software development.[20]

The report first positions "Mythic Engineering" as a new paradigm that transcends "Vibe Coding." Subsequently, it will elaborate on the architecture of Viking Code: a command system stewarded by an "Overlord" and executed by a team of expert "Centurion" agents. This design deconstructs complex projects into manageable, independent workflows and provides an unparalleled level of guidance, context management, and control through a Domain-Specific Language (DSL) centered on Pipelines and Recipes, optimized for agent use, thereby moving Mythic Engineering from concept to actionable reality.

1. The Evolution from Vibe Coding to Mythic Engineering

1.1 Defining Vibe Coding

Vibe Coding, popularized by OpenAI co-founder Andrej Karpathy, describes a development practice where developers describe requirements to a Large Language Model (LLM) in natural language, the AI generates the code, and the developer judges correctness by running the result rather than reading the code.[20] Its core characteristic is the developer's conscious surrender of line-by-line understanding of the code.

"If an LLM wrote every line of your code but you reviewed and understood all of it, that's not vibe coding - that's using an LLM as a typing assistant." — Simon Willison[20]

1.2 The Modern AI Programming Tool Landscape

AI-Native IDEs

Such as Cursor, offering Composer and Agent modes, suited for professional developers for daily coding and fine-grained control.[20]

Terminal Agents

Such as Claude Code, a pure terminal agent with no graphical interface, excelling at deep understanding, navigation, and refactoring of large, complex codebases (millions of lines).[20]

App Generators

Such as Bolt.new, capable of generating full-stack web applications directly from natural language descriptions in the browser, ideal for rapid prototyping.[20]

Autonomous Agents

Such as Devin, positioned as a fully autonomous AI software engineer capable of handling the entire software development lifecycle.[20]

1.3 A New Paradigm: Agentic Engineering and Mythic Engineering

By 2026, the concept of AI programming has evolved far beyond Vibe Coding. It has grown into a spectrum encompassing multiple modalities, including:

Agentic Engineering

Proposed by Karpathy in February 2026, seen as the "disciplined version" of Vibe Coding. It emphasizes human oversight and quality control, with the developer's role shifting from "prompt engineer" to "architect" and "orchestrator."[20][42]

Harness Engineering

The core is "human at the helm + agent execution." It's not about optimizing the model itself, but building a complete set of constraint mechanisms, feedback loops, and workflow management systems around the AI agents.[42]

Ralph Wiggum Loop

Putting the AI in a loop, executing repeatedly until all checklist items in the requirements document are completed. Progress is persisted via Git commits, supporting unattended execution.[42]

1.4 The Architect's Design Philosophy

Terminal-Centric

Establishes terminal agents like Claude Code as the core of Viking Code, considering the terminal to be the most direct and controllable interface for engineers.[20]

Command, Don't Do

The human's role is to design the system, formulate strategy, and oversee execution, not to get bogged down in every code detail.

Decoupling and Control

Breaks down complex projects through mandatory separation of concerns and provides complete, traceable execution logs.

2. Viking Code Architecture Overview

2.1 Core CLI Architecture

Viking Code features a layered, API-first architecture designed to provide a clear human-machine interface.

CLI Layer (Human-Facing)

  • • Clear command structure: viking init, viking plan
  • • YAML-based "Recipe" system
  • • CLI patterns inspired by Git and Docker

Control Plane (Agent-Facing)

  • • /agents API: Agent discovery and control
  • • /tasks API: Task assignment and status updates
  • • Event Bus: System-wide event propagation

2.2 Control Plane: The AI Agent

The "Overlord"

The central intelligence of Viking Code, acting as the central nervous system. Designed to be intelligent, proactive, and highly controllable.

Task Decomposition & Orchestration

Reads recipe files, breaking down complex epics into ordered tasks

Agent Delegation

Assigns tasks to the most suitable Centurion agent

Supervision & Recovery

Monitors agent health, reassigning tasks upon failure

State Management

Maintains a centralized task state database

Security Oversight

Monitors suspicious activity, integrates with OPA policy engine

2.3 Data Plane: Connections and Integrations

Agent Pool

Each Centurion is an independent, specialized AI agent with its own LLM backend, isolated filesystem sandbox, and a small internal knowledge base. They communicate with the Overlord via gRPC or RESTful APIs.

Example Agent Types:

Code Generator Test Writer Documentation Writer Security Auditor

External Integrations

3. Core Domain Deep Dive

3.1 Epics

Definition

An epic is a human-readable, high-level description of a feature or requirement, typically a collection of phrases.

Example

"Implement email verification for the user module, including sending verification emails, validating tokens, and updating user status."

Manifestation in Viking

The viking init command guides the user to create an epic and initializes it into the project structure.

3.2 Recipes

Definition

A recipe is a machine-readable, version-controlled execution plan for an epic. It is a YAML file that deconstructs an epic into a series of independent, idempotent tasks.

Idempotency

Result is the same whether executed once or a hundred times

Atomicity

Tasks are as independent as possible, with clearly defined inputs and outputs

Explicit Dependencies

Cross-task dependencies must be explicitly declared in the recipe file

Example Structure

epic: "feat(user): add email verification"
version: 1
tasks:
  - id: analyze_prd
    description: "Analyze existing user module architecture and determine integration points."
    agent_type: "architect"
    inputs:
      - "source_code"
      - "product_requirements_document"
    outputs:
      - "architecture_design_document"
      - "integration_points"
    estimated_tokens: 15000
    environment:
      - "sandbox"
    retry_policy:
      - "on_failure"
      - max_attempts: 3

3.3 Agents

Definition

An Agent is an AI program given a specific goal (task) and authorized to act autonomously to achieve it. Its behavior is entirely constrained by the boundaries defined by its Persona and available Tools.

Centurions

The expert-level agents in Viking. Each Centurion is proficient in a specific function, such as "Backend Engineer," "Frontend Engineer," "Test Engineer," "Documentation Writer," or "Security Auditor."

Persona

Defines the agent's expertise, communication style, and core principles

"You are a Rust backend expert with 10 years of experience, focused on security and performance."
Tools

Functions authorized for the agent to use

run_tests, read_file, write_file
Context

Task-specific information like recipes, relevant code snippets, or previous execution logs

Memory

Optional persistent vector storage for retaining knowledge across multiple tasks

3.4 Context

Definition

Context is the lifeblood of all AI interactions. Viking Code's context management system is designed to provide the right information to the agent at the right time.

Project Level

The VIKING.md file defines project-wide conventions, tech stack, and high-level goals[20]

Epic Level

The recipe file itself provides the scope and constraints of the epic

Task Level

Each agent receives a packaged context including relevant source files, outputs from prior tasks, and user instructions

Session Level

The agent's "short-term memory," existing for the duration of the task

Global Level

Relevant information from an internal knowledge base (RAG), such as company API design standards documents

Context Packing

The system automatically packages relevant code, documentation, and prior outputs, intelligently pruning irrelevant information to maximize information density within the limited context window.

Knowledge Graph

Viking maintains a project knowledge graph linking entities like files, functions, API endpoints, and bugs, enabling highly precise context retrieval.

4. Advanced Feature Implementation

4.1 CLI DSL Design

Concept

A Domain-Specific Language (DSL) for agent interaction, designed to maximize clarity and consistency while minimizing the token count required by agent LLMs.

Design Principles
  • • Imperative: Uses active voice imperatives
  • • Noun-Verb Structure: Resource first, action second
  • • Strongly Typed Flags: GNU-style long format flags
  • • Pipe Native: Output is easily pipeable
Advantage Examples
✓ viking plan ✗ viking plan_request
✓ vikcent run ✗ viking task_run
✓ --recipe=viking.yml ✗ -r viking.yml

Example Workflow

# Initialize a new epic
viking init --name "user-mgmt" --description "Implement user management epic."

# Generate a recipe based on the epic
viking plan --epic user-mgmt.md --recipe viking.yml --force

# Inspect the recipe
cat viking.yml

# Start execution
vikcent run --recipe viking.yml --agent architect --task analyze_prd

# Monitor progress
viking task status --task analyze_prd

# View agent logs
viking task logs --task analyze_prd --follow

# If architecture changes, update recipe and restart
viking plan --epic user-mgmt.md --recipe viking.v2.yml
vikcent run --recipe viking.v2.yml

# Complete the epic
viking complete --epic user-mgmt --status done --summary "Epic completed successfully."

4.2 Port System

Definition

A Port is a named input or output point that allows structured data to be exchanged between tasks, enabling loose coupling.

Input Port Examples
  • • source_code: Access to the codebase
  • • product_requirements: Product requirements document
  • • architecture_document: Architecture design document
Output Port Examples
  • • architecture_design_document: Path to design document
  • • integration_points: List of integration points
  • • test_results: Test execution results

Connection Mechanism

Connection definitions in the recipe file specify how an output port of one task connects to an input port of another.

Task A Output
outputs:
  - "architecture_design_document"
Task B Input
inputs:
  - "architecture_design_document"

4.3 Audit System

Definition

Audit is one of the core principles of Viking. All operations requiring human judgment are explicitly logged, flagged, and made available for review.

Logs

All agent activity (prompts, tool calls, responses, file edits) is logged in structured JSON format to persistent logs. The viking task logs command retrieves these logs.

Audit Trails

A separate, immutable audit log records all state transitions, task assignments, and completed changesets, similar to git's commit history, but for AI operations.

Human Review Gates

Certain tasks (like deploy or merge) can be flagged as requiring human approval. The Overlord will pause execution and wait for human confirmation via the viking approve command.

Security Logs

Integration with Open Policy Agent (OPA) can generate security audit logs recording the rationale for access control decisions.

Compliance Reports

The viking audit report --format=json command can generate a complete compliance report detailing the execution of each task, tokens used, and changes made.

Traceability

Can be used to track "who (which agent) did what and when," providing complete traceability.

5. Evolution Roadmap

5.1 From Python Rewrite to the First Agentic MVP

Goal

Build a Minimum Viable Product supporting AI agents, pluggable into existing toolchains.

Tech Stack Selection

Rewrite the existing prototype from Python to Go. Go has inherent advantages in concurrency, API services, and CLI tool development.

Agent Integration

Integrate an existing open-source terminal agent like Aider or Opencode as a "Centurion-01" expert.[25]

Core Commands

Implement the viking init and viking plan commands to create epics and recipes in YAML format.[58]

Task Execution

Implement the vikcent run command to delegate tasks to the integrated agent and stream its output.

Success relies on early integration of existing, proven tools (like Claude Code), gradual expansion via the agent architecture, and ultimately differentiation through innovation in context management and security.

Conclusion: The Future of the CLI

Viking Code is not the final destination, but an evolutionary intermediate step. It points directly towards a future possibility: a truly autonomous software engineering agent—given just a specification and a Viking Code server, capable of delivering a production-ready system. The Mythic CLI is the crucial cornerstone paving the way to this future: an auditable, scalable, and fully controlled platform commanded by a human strategic mind and executed by expert agents.

The design principles of Viking Code—decoupling, idempotency, explicit context, and machine-readable recipes—represent the inevitable direction for the evolution of AI programming tools. As Large Language Models continue to advance, tools for controlling and managing the "digital workforce" they generate will become as important as the models themselves.[47] Viking Code is designed precisely for building and managing this future workforce.

The ultimate goal of Viking Code is to empower human architects to skillfully navigate the exponential complexity introduced by AI-generated code. By placing the human in the role of designer and supervisor, rather than executor, it directly addresses the "black box" and maintainability challenges posed by Vibe Coding.[22] It solidifies emerging methodologies like SDD and the Ralph Wiggum Loop into executable processes by introducing "Recipes" and explicit task boundaries.[44]

By providing audit logs, human gating, and policy-as-code, it meets enterprise-level needs for security and compliance.[56] Therefore, Viking Code is not just another tool. It is a redefinition of the act of "programming" itself, in an era where software engineering is increasingly agent-driven. It is a bridge to autonomous software engineering, offering a clear, controllable, and scalable framework for the future of AI-driven development teams.[52]
